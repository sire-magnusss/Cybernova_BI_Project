SECTION 1 — FUNCTIONAL REQUIREMENTS (FR1–FR12)

FR1 — Generate ≥1,000-row synthetic IIS W3C log dataset
Status: FULLY MET

Evidence:
- generate_logs.py produces raw IIS-style logs and an enriched CSV.
- config.yaml drives date range (2026-01-01 to 2026-05-15), base 500 daily
  requests with weekday/weekend variation, monthly growth at 8%, and a 35%
  random fluctuation — this produces a dataset well above 1,000 rows.
- All required fields are present: timestamps (date + time), Southern African IPs
  (country-specific prefixes like 102.128.x.x for Botswana), service URIs,
  HTTP methods, status codes, bytes, user agents.
- Realistic patterns are implemented: peak-hour clustering via
  random_time_for_human_session() (42% morning, 38% afternoon), bot traffic
  with off-peak weighting via random_time_for_bot_session(), and 404 errors
  via the anomaly system (e.g., "Broken AI Campaign Link" on 2026-04-03).
- Campaign-driven traffic spikes (multipliers up to 2.0x) add further realism.

No gaps.

FR2 — Ingest and preprocess log file: parse all fields, resolve IPs to countries,
      handle malformed records, load clean DataFrame, report records
      processed/cleaned/dropped
Status: MOSTLY MET — one reportable gap

Evidence:
- load_data() reads the Excel workbook via a temp-file copy (OneDrive lock
  workaround), joins the enriched CSV on a composite key, parses timestamps,
  casts boolean columns, and sorts by timestamp.
- IP-to-country resolution is handled at generation time via the static country
  prefix mapping in generate_logs.py — this satisfies the "static lookup"
  constraint stated in the Constraints section.
- Malformed records are handled via try/except blocks; missing values are filled
  or coerced rather than crashing.

Gap:
- The SRS requires reporting "records processed, cleaned, and dropped" to the
  user. This report does not appear in the dashboard or as a log message.
  The only count shown is the row count chip in the filter context bar, which
  reflects the filtered dataset, not the raw-load outcome. A one-line summary
  after data load (e.g., "Loaded 45,320 rows; 12 malformed records dropped")
  would close this gap.

FR5 — Simulate a real-time log feed that detects new entries and refreshes all
      dashboard metrics without manual reload.
Status: MET IN SPIRIT — mechanism honest and correctly labelled

Evidence:
- render_live_section() is decorated with @st.fragment(run_every=3), which
  causes Streamlit to re-execute the fragment every 3 seconds automatically.
- st.session_state["live_log"] holds a ring buffer capped at 300 records; one
  row is ingested per cycle by advancing a pointer through the dataset.
- The stream label throughout reads "STREAMING FROM LOG STREAM", and all live
  chart subtitles carry the disclaimer "Simulated live view from timestamped
  synthetic IIS logs" — no false production claims are made.

Note on interpretation:
- The SRS says "file-watch or polling mechanism that detects new entries". The
  implemented approach simulates this via dataset cycling rather than watching a
  file. Given Constraint 1 (no real server logs) this is the only viable
  approach and is correctly presented as simulation. A tutor reading the spec
  against the code should accept this as a justified design decision, provided
  the CRISP-DM documentation explains the reasoning.

SECTION 2 — NON-FUNCTIONAL REQUIREMENTS (NFR1–NFR8)
────────────────────────────────────────────────────────────────────────────────

NFR1 — Dashboard loads within 3 seconds; filter interactions within 1 second.
Status: LIKELY MET

@st.cache_data on load_data() means the Excel file is read once and cached in
memory for subsequent interactions. The OneDrive temp-file copy prevents
repeated locking delays. Filter interactions operate on the already-loaded
DataFrame in memory, which should be well under 1 second for datasets up to
50,000 rows. This cannot be formally verified without timing measurements, but
the architecture is appropriate.

────────────────────────────────────────────────────────────────────────────────

NFR2 — Log data strictly conforms to IIS W3C format. Country distribution
       reflects realistic Southern African proportions; ≥30% from Botswana.
Status: FULLY MET

config.yaml explicitly sets Botswana: 0.32 (32%), which satisfies the ≥30%
requirement. South Africa follows at 20%, with the remaining 48% spread across
eight other SADC nations — proportions that broadly reflect realistic regional
internet traffic patterns.

IIS W3C fields generated: date, time, ip_address, uri, method, http_status,
bytes_sent, user_agent. All standard IIS Extended Log Format fields are covered.

────────────────────────────────────────────────────────────────────────────────

NFR3 — Fixed random seed ensures full dataset reproducibility.
Status: FULLY MET

config.yaml sets random_seed: 333 at project level. The seed is loaded by
generate_logs.py at startup, ensuring identical datasets across runs.

────────────────────────────────────────────────────────────────────────────────

NFR4 — Consistent UI: colour scheme, ≥11pt font, appropriate information
       density per audience.
Status: FULLY MET

Design tokens (NAVY, BLUE, CYAN, GREEN, AMBER) are defined at the top of
app_html.py and applied consistently through all card, chart, and text styling.
Base font size is 11px or higher throughout (11px–31px range in KPI cards).
Information density is appropriately calibrated: the Executive dashboard
presents 5 KPIs and a decision brief; Sales and Marketing show deeper evidence
tables, action queues, and segment detail. The professional login page and
branded hero cards add presentation polish above typical academic deliverables.

────────────────────────────────────────────────────────────────────────────────

NFR5 — Modular codebase: separate modules for generation, ETL, analysis, and UI.
       Inline documentation throughout. Built for monthly re-execution.
Status: PARTIALLY MET

Separation achieved:
- generate_logs.py: data generation, well-documented with clear function
  docstrings throughout.
- synthetic_log_studio.py: generation UI interface.
- config.yaml: all parameters externalised — changing dates, country weights,
  campaigns, or anomalies requires only config edits, not code changes. This
  directly supports monthly re-execution.

Gap:
- The SRS requires separate modules for ETL, analysis, and UI. All three are
  combined in app_html.py (2,683 lines): load_data() (ETL), calculate_*()
  functions (analysis), and render_*/show_*() functions (UI). By the SRS
  definition, these should be in separate files (e.g., etl.py, analytics.py,
  ui.py or similar). At 2,683 lines, the single file is navigable but does not
  satisfy the modularity requirement as written.
- Inline documentation is strong in generate_logs.py but sparse in app_html.py
  — most functions have no docstring and section comments replace formal
  documentation.

────────────────────────────────────────────────────────────────────────────────

NFR6 — Pipeline handles up to 50,000 log entries without restructuring.
Status: LIKELY MET

Core data operations use pandas vectorised methods throughout. The ring buffer
caps at 300 records to avoid unbounded memory growth in the live section. No
Python loops operate over the full dataset. @st.cache_data prevents redundant
re-reads. Standard pandas on 50,000 rows should perform within acceptable
Streamlit response times on a modern machine.

────────────────────────────────────────────────────────────────────────────────

NFR7 — Deployable locally or via Streamlit Community Cloud from requirements.txt.
       No proprietary BI platforms.
Status: FULLY MET

requirements.txt is present. The stack (Streamlit, Plotly, Pandas, NumPy,
ReportLab, openpyxl, PyYAML) is entirely open-source. No Power BI, Tableau,
or proprietary tools are used. The only deployment blocker for Streamlit Cloud
is the OneDrive data path — the Excel file path "data/output/cybernova_web_logs.xlsx"
is relative and would need the data directory committed or regenerated via the
Synthetic Log Studio before cloud deployment.

────────────────────────────────────────────────────────────────────────────────

NFR8 — All data entirely synthetic. No real personal data. Documented to
       professional standards.
Status: FULLY MET

All IP addresses are algorithmically constructed from country-specific prefix
ranges with no connection to real users. User agents are templates, not scraped.
Session IDs are MD5 hashes of synthetic inputs. The dashboard, login page, and
all live chart subtitles carry "Simulated live view from timestamped synthetic
IIS logs" and "CET333 Product Development prototype" disclaimers. generate_logs.py
is fully documented with module-level docstring and per-function documentation.

   stated.


WHERE WORK REMAINS (PRIORITY ORDER)

The following items are gaps against the SRS that should be addressed before
client sign-off or submission:

PRIORITY 1 — FR12 (Descriptive Statistics + Correlation)
This is the only requirement rated "partially met" and represents material
analytical work missing from the prototype.

a. Add a descriptive statistics summary table to at least one dashboard.
   Suggested location: Sales (per-service) or Marketing (per-country).

   instead of day_of_week. Suggested title: "When is each service most

c. Add the correlation statistic itself (e.g., Pearson r or a written
   interpretation) as an insight note below the heatmap.


PRIORITY 2 — FR10 (Demo vs Event separate quantification)
Action items:
a. Add two separate KPI cards or a two-row mini-table in the Sales dashboard:
   "/scheduledemo.php requests: X" and "/events.php requests: Y".
b. Add "Warm leads as % of total traffic" as a dedicated KPI card or delta note.


PRIORITY 3 — FR6 (Underperforming regions flag)
Action items:
a. In the Executive or Marketing geographic section, flag countries with fewer
   than a defined threshold of requests (e.g., < 5% of total) as
   "Below expansion target" using an amber indicator. This is a small

PRIORITY 4 — FR2 (ETL drop report)

OVERALL VERDICT
─────────────────
This is a strong prototype that substantially exceeds the SRS baseline. The
visual design, live streaming architecture, role-based access, forecast charts,
insight cards, and PDF export represent work well above a standard CET333
submission. The core analytical pipeline (ETL, segmentation, geographic BI,
AI traction analysis) is solid and well-evidenced throughout.

The two material gaps — missing descriptive statistics (FR12) and missing
demo/event separate quantification (FR10) — are analytically significant and
should be addressed before sign-off. FR12 in particular is the kind of gap
that an academic evaluator will look for when checking whether the system
performs genuine BI analysis versus only visualisation.

If FR12, FR10, and the FR6 flag are addressed, the system meets all twelve
functional requirements and all eight non-functional requirements either fully
or with documented justification. The deliverable would be ready for client
