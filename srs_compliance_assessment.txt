CyberNova BI Project - SRS Compliance Assessment
Generated: 2026-05-06

Purpose
This document consolidates the current project structure and assesses the CyberNova BI dashboard against the requirements listed in srs.txt. The assessment uses three statuses:

PASS    - Requirement is substantially implemented.
PARTIAL - Requirement is present but incomplete, hardcoded, not fully connected, or weakly evidenced.
FAIL    - Requirement is not currently evidenced in the implementation.


1. Current Folder / Project Structure

Main dashboard application:
- cybernovaapp.py
  Streamlit entry point. Handles login, role routing, dashboard selection, global filter drawer, four tabs, data loading, shared CSS, and shared overview widgets.

Role-specific dashboard modules:
- sales_views.py
  Sales dashboard views for Analytics, Forecasting, and Data & Export.

- marketing_views.py
  Marketing dashboard views for Analytics, Forecasting, and Data & Export.

- executive_views.py
  Executive / Management dashboard views for Analytics, Forecasting, and Data & Export.

Data generation and validation:
- generate_logs.py
  Synthetic IIS web log generator and enrichment pipeline. Produces raw IIS logs, enriched BI logs, Excel workbooks, summaries, potential customer outputs, forecasts, and quality summaries.

- validate_dataset.py
  Dataset validation script. Checks request counts, countries, services, status codes, segment distribution, warm lead rate, bot ratio, campaigns, anomalies, and daily request summaries.

Supporting modules:
- exports.py
  PDF, CSV, Excel, KPI summary, country summary, and methodology export helpers.

- live_feed.py
  Simulated live feed writer for generating incremental synthetic traffic rows.

Configuration and requirements:
- config.yaml
  Controls seed, date range, traffic distribution, country proportions, services, campaigns, anomalies, and output paths.

- requirements.txt
  Python dependencies for local or Streamlit deployment.

Data outputs:
- data/output/cybernova_iis_raw_logs.csv
- data/output/cybernova_enriched_logs.csv
- data/output/cybernova_web_logs.xlsx
- data/output/cybernova_decision_engine_dataset.xlsx
- data/output/generation_summary.csv

3. Functional Requirements Assessment

FR1 - Generate >=1,000-row synthetic IIS W3C log dataset
Status: PASS
Evidence:
- Generated enriched dataset contains 60,551 requests.
- Raw IIS CSV exists at data/output/cybernova_iis_raw_logs.csv.
- Enriched CSV exists at data/output/cybernova_enriched_logs.csv.
- Dataset includes timestamps, Southern African IP-style addresses, URIs, HTTP methods, status codes, bytes, response times, and user agents.
- Validation shows realistic status distribution, bot traffic, campaigns, anomalies, 404s, and country proportions.

Concern:
- Raw CSV contains IIS-style fields, but it is not a literal W3C log text file with W3C header lines. If the marker expects strict W3C Extended Log Format text, this should be strengthened or documented.


FR2 - Ingest and preprocess log file; parse fields, resolve countries, handle malformed records, report processed/cleaned/dropped
Status: PARTIAL
Evidence:
- generate_logs.py creates enriched BI-ready fields from raw request data.
- load_data() in cybernovaapp.py loads the enriched CSV and parses timestamp/date fields.
- Country, city, region, service group, segment, quality, risk, and funnel fields exist in the enriched dataset.

Gap:
- The dashboard does not clearly show records processed, cleaned, and dropped.
- There is no separate visible ETL report panel proving malformed record handling.
- App loads a prepared enriched CSV rather than visibly running a modular ETL step.


FR3 - Perform EDA: univariate, bivariate country x service, hourly/daily time series decomposition, anomaly detection, commentary

6. Main Issues to Fix Before Final Submission

1. Apply all global filters properly.
   Current app applies date and market, but service, segment, outcome, and HTTP status need to filter the DataFrame before every tab renders.

2. Add an explicit HTTP status filter.
   The SRS specifically requires HTTP status filtering.

3. Make real-time refresh evidence stronger.
   Connect live_feed.py or a polling file source to the dashboard so new rows update metrics and visuals from data, not only animated counters.

4. Replace or clearly label hardcoded/mock dashboard content.
   Several maps, forecasts, insight cards, rankings, and KPI values use fixed values or seeded random data. For final SRS compliance, key visuals should be data-driven from the filtered enriched dataset.

5. Add a formal statistics/correlation tab section.
   Include mean, median, standard deviation, totals by country/service/time period, plus time-of-day vs service-type analysis.

6. Separate warm lead reporting for /scheduledemo.php and /event.php.
   The app should show count, country split, time trend, and rate as percentage of total traffic for each route separately.

7. Add ETL quality reporting.
   Show records processed, cleaned, dropped, invalid rows, missing fields, duplicate sessions, and quality score in a visible dashboard/export section.

8. Fix encoding corruption.
   Many files display mojibake such as "â€¢", "âš", and "ðŸ". This can make the dashboard and documentation look broken.

9. Validate performance.
   Time first load and filter interactions. The 70 MB enriched CSV may be acceptable locally but should be measured against NFR1.

10. Resolve SRS wording mismatch.
   The SRS says "three-tab dashboard", but the current app has four tabs per role. Either change the SRS to four tabs or explain the fourth tab as an enhancement.


7. Overall Verdict

Overall status: PARTIAL PASS

The project is strong and already demonstrates the main CyberNova BI concept: synthetic IIS web traffic, enriched analytics data, three role-based dashboards, four tabs per role, many visualisations, exports, and a polished Streamlit interface.

However, the project should not yet be described as fully SRS-compliant because some requirements are only visually represented or partially wired. The most important compliance gaps are global filtering, real-time data refresh, explicit ETL quality reporting, descriptive statistics/correlation analysis, and replacing hardcoded/mock visuals with filtered dataset-driven results.

Priority for final improvement:
1. Fix global filters.
2. Add HTTP status filter.
3. Add stats/correlation section.
4. Make warm lead tracking separate for demo and event pages.
5. Strengthen real-time refresh from actual appended data.
6. Reduce hardcoded/mock values in core evidence charts.
7. Fix encoding issues.
